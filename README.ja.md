# Karpathy Method Skills

AI エージェントを「速いけれど雑な実行者」から、「前提を確認し、過剰に作らず、検証まで回す信頼できる共同作業者」に近づけるためのスキルです。

このスキルは、Andrej Karpathy 氏が指摘した LLM コーディングの典型的な失敗、つまり「勝手な仮定」「過剰設計」「関係ない変更」「検証不足」を減らすための行動原則を、再利用可能な skill / `CLAUDE.md` として整理したものです。

[English](./README.md) | [Chinese](./README.zh.md)

## 概要

このスキルの本質は、AI にすぐ生成させるのではなく、まず次の4つを徹底させることです。

- 前提や曖昧さを明示する
- 必要最小限の解決策を選ぶ
- 関係ない場所を勝手に変更しない
- 成功条件と検証方法を決めてから実行する

もともとはコーディング向けのガイドラインですが、実際には非エンジニアの仕事にも使えます。多くの AI 失敗はコード固有ではなく、曖昧な目的、隠れた前提、過剰な一般化、検証不足から起きるためです。

## コーディングでの使い方

次のような場面に向いています。

- バグ修正
- 既存機能への小さな変更
- リファクタリング
- PR レビュー
- テスト追加
- AI が勝手に大きく作りすぎるのを防ぎたい時

例:

```text
$karpathy-guidelines
このバグを最小変更で直して。まず前提、成功条件、検証方法を整理して。
関係ないコードやコメントは変更しないで。
```

期待できる効果:

- 差分が小さくなる
- 不要な抽象化や過剰設計が減る
- 関係ないファイル変更が減る
- テストや確認手順が明確になる
- レビューしやすい PR になる

## コーディング以外での使い方

このスキルは、エンジニア以外にも有効です。AI に調査、分析、文書作成、戦略整理を依頼する時にも、同じ原則が効きます。

向いている用途:

- 戦略メモや経営向け資料のレビュー
- 市場調査、競合分析、ファイナンス調査
- PRD、事業計画、施策案の整理
- 採用基準、評価基準、業務プロセスの設計
- 法務、リスク、コンプライアンス文書の確認
- 記事、SNS、メッセージの構成改善

例:

```text
$karpathy-guidelines
この戦略メモをレビューして。すぐ書き直さず、まず隠れた前提、
不足している根拠、意思決定基準、最小限で有効な提案を整理して。
```

期待できる効果:

- 前提と論点が見えやすくなる
- 生成物が無駄に長くなりにくい
- 「それは違うかも」という健全な反論が出やすい
- 事実、解釈、提案が分離される
- あとから検証、修正、再利用しやすくなる

## 使い方のコツ

依頼するときは、単に「作って」ではなく、次のように頼むと効果が出やすくなります。

```text
まず前提を確認して。
曖昧な点があれば質問して。
最小限の解決策を出して。
成功条件を定義して。
検証方法も書いて。
不要な拡張や関係ない改善はしないで。
```

## 一言でいうと

Karpathy Method Skills は、AI に「賢く速く作らせる」だけでなく、「勝手に進めすぎない、余計なことをしない、最後にちゃんと確認する」ための安全運転モードです。
